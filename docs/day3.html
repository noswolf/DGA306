กระบวนการ ETL คืออะไร?

ETL ย่อมาจาก Extract · Transform · Load คือขั้นตอนหลักในการนำข้อมูลเข้า Data Warehouse คลิกที่แต่ละขั้นตอนเพื่อดูรายละเอียด

📤
E — Extract
ดึงข้อมูลดิบจากแหล่งข้อมูลต่าง ๆ ไม่ว่าจะเป็น ฐานข้อมูล SQL, ไฟล์ CSV, JSON หรือ API แล้วนำไปเก็บไว้ใน Staging Area (พื้นที่กักกันชั่วคราว)
👆 คลิกเพื่อดูรายละเอียด
⚙️
T — Transform
แปลง ทำความสะอาด และจัดรูปแบบข้อมูล เช่น เติมค่าที่หายไป แปลงรูปแบบวันที่ ลบข้อมูลซ้ำ และตรวจสอบความถูกต้องก่อนโหลดเข้า Warehouse จริง
👆 คลิกเพื่อดูรายละเอียด
📥
L — Load
นำข้อมูลที่ผ่านการ Transform แล้วโหลดเข้าสู่ Data Warehouse จริง โดยโหลด Dimension Tables ก่อนเสมอ แล้วจึงโหลด Fact Tables เพราะ Fact ต้องอ้างอิง Dimension
👆 คลิกเพื่อดูรายละเอียด
🏪
Staging Area
พื้นที่กักกันข้อมูลชั่วคราวก่อน Transform เปรียบเหมือน "ห้องกักกัน" ที่ข้อมูลดิบรอตรวจสอบ ช่วยแยกระบบต้นทางออกจาก Warehouse และรองรับการ Audit
👆 คลิกเพื่อดูรายละเอียด
🗄️
Data Warehouse
คลังข้อมูลกลางที่เก็บข้อมูลที่ผ่านการทำความสะอาดแล้ว ประกอบด้วย Fact Tables (ข้อมูลเชิงตัวเลข) และ Dimension Tables (บริบท เช่น ลูกค้า สินค้า วันที่)
👆 คลิกเพื่อดูรายละเอียด

การออกแบบ Schema สำหรับ Data Warehouse

เลือกดู Schema แต่ละแบบ แล้วทดสอบความเข้าใจด้วยแบบทดสอบด้านล่าง

Star Schema — โครงสร้างดาว

Fact Table อยู่ตรงกลาง เชื่อมต่อตรงกับ Dimension Tables โดยไม่ผ่านตารางกลาง ง่ายและค้นหาเร็ว

✅ ข้อดี
  • Query ง่ายและเร็ว (Join น้อย)
  • เหมาะกับ BI Dashboard เช่น Tableau, Power BI
  • นักวิเคราะห์ธุรกิจเข้าใจง่าย
⚠️ ข้อจำกัด
  • มีข้อมูลซ้ำบ้างใน Dimension
  • ใช้พื้นที่จัดเก็บมากกว่า
  • ไม่เหมาะกับงาน Write-heavy
💡 แนะนำใช้ Star Schema เมื่อ: ต้องการ Dashboard ที่โหลดเร็ว ตอบโจทย์ธุรกิจชัดเจน
Snowflake Schema — โครงสร้างเกล็ดหิมะ

Dimension Tables ถูกแตกย่อยออกเป็นตารางย่อยๆ (Sub-Dimension) เพื่อลดข้อมูลซ้ำ แต่ต้อง Join มากขึ้น

✅ ข้อดี
  • ข้อมูลซ้ำน้อยกว่า (Normalized)
  • ประหยัดพื้นที่จัดเก็บ
  • เหมาะกับ Warehouse ขนาดใหญ่
⚠️ ข้อจำกัด
  • Query ซับซ้อน ต้อง Join หลายชั้น
  • นักวิเคราะห์เข้าใจยากกว่า
  • เครื่องมือ BI บางตัวต้องการ Query ซับซ้อน
💡 แนะนำใช้ Snowflake เมื่อ: ข้อมูลขนาดใหญ่มาก ต้องการ Normalize อย่างเคร่งครัด หรือสร้าง Analytical Model ซับซ้อน
Star vs Snowflake — เปรียบเทียบแบบเข้าใจง่าย
ด้านที่เปรียบ Star Snowflake
โครงสร้าง เรียบง่าย — Fact เชื่อมตรง Dim ซับซ้อน — Dim แตกย่อยหลายชั้น
ความเร็ว Query ✅ เร็วกว่า (Join น้อย) ⏳ ช้ากว่า (Join มาก)
การใช้พื้นที่ ใช้มากกว่า (มี Redundancy) ✅ ใช้น้อยกว่า
ความง่ายต่อผู้ใช้ ✅ นักวิเคราะห์เข้าใจง่าย ต้องการความรู้ด้าน DB มากกว่า
เหมาะสำหรับ BI Dashboard, Report รายงาน Data Warehouse ขนาดใหญ่, ETL ซับซ้อน
ทดสอบความเข้าใจ
1. หน่วยงานต้องการสร้าง Dashboard รายวันให้ผู้บริหาร ต้องการ Query เร็ว ควรเลือก Schema แบบใด?
2. Fact Table ควรเก็บข้อมูลประเภทใด?
3. ใน ETL ควรโหลดตารางใดก่อนเมื่อนำข้อมูลเข้า Warehouse?

Wide Format vs Long Format

รูปแบบข้อมูลสองแบบที่พบบ่อยในการวิเคราะห์ข้อมูล เข้าใจความแตกต่างเพื่อเลือกใช้ให้ถูกต้อง

📖 แนวคิดพื้นฐาน

ดูตัวอย่าง: ข้อมูลจำนวนสัตว์ป่าในอุทยานต่างๆ — ข้อมูลชุดเดียวกัน จัดรูปแบบต่างกัน

⬛ Wide Format
อุทยาน เสือโคร่ง ช้าง กวาง
เขาใหญ่ 20 150 420
ดอยสุเทพ 5 80 210
อ่าวพังงา 0 30 95

แต่ละชนิดสัตว์ = 1 คอลัมน์
1 แถว = 1 อุทยาน

⇄
📋 Long Format
อุทยาน ชนิดสัตว์ จำนวน
เขาใหญ่ เสือโคร่ง 20
เขาใหญ่ ช้าง 150
เขาใหญ่ กวาง 420
ดอยสุเทพ เสือโคร่ง 5
... ... ...

1 คอลัมน์สำหรับชนิดสัตว์
1 แถว = 1 การสังเกต

🔑 กฎจำง่าย: Wide = กว้าง (คอลัมน์มาก) | Long = ยาว (แถวมาก) — เครื่องมือวิเคราะห์เช่น Python (pandas), R, Tableau มักต้องการ Long Format
ใช้ Format ไหน ตอนไหน?
⬛ Wide Format เหมาะสำหรับ
  • นำเสนอในรายงาน / Excel
  • มนุษย์อ่านแล้วเข้าใจง่าย
  • ข้อมูลสรุปรวม (Pivot Table)
  • ป้อนข้อมูลเข้าระบบ
📋 Long Format เหมาะสำหรับ
  • วิเคราะห์ด้วย Python/R/SQL
  • สร้าง Visualization ด้วย Plotly, Seaborn
  • Machine Learning Features
  • จัดเก็บใน Database / Data Warehouse
แปลงด้วย Python — melt() และ pivot()
Wide → Long ด้วย melt()
# Wide → Long import pandas as pd wide_df = pd.DataFrame({ 'อุทยาน': ['เขาใหญ่', 'ดอยสุเทพ'], 'เสือโคร่ง': [20, 5], 'ช้าง': [150, 80] }) long_df = wide_df.melt( id_vars=['อุทยาน'], var_name='ชนิดสัตว์', value_name='จำนวน' )
Long → Wide ด้วย pivot()
# Long → Wide wide_back = long_df.pivot( index='อุทยาน', columns='ชนิดสัตว์', values='จำนวน' ).reset_index() # ลบ MultiIndex ที่คอลัมน์ wide_back.columns.name = None print(wide_back)
